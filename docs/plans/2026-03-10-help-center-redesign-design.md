# 帮助中心改版设计

## 概述

当前帮助中心仍沿用早期的单层目录加长文结构，内容覆盖范围与产品现状已经脱节。前端已经新增数据库配置、WAR 包管理、License 备份导入、数据库备份恢复、个人中心等功能，但帮助中心仍主要覆盖早期的 POS、移动设备、Linux、工作台和管理员基础说明，且信息组织方式不适合继续扩展。

本次改版的目标是把帮助中心升级为“教程 + 索引结合”的单页双层导航页面：用户先按模块定位，再在模块内部通过二级目录快速找到目标内容，最后查看结构化教程内容。

## 目标

- 完整覆盖当前系统已有的主要功能模块
- 将帮助中心从“长文档”改为“模块切换 + 模块内目录”的双层导航结构
- 让帮助中心同时满足新手入门和老用户快速检索两类场景
- 优化 UI 展示，使信息层级更清晰、卡片化更强、阅读负担更低
- 保持现有 `/help` 路由与导航入口不变，控制改动范围

## 非目标

- 不引入后端帮助内容管理能力
- 不新增帮助中心搜索接口
- 不拆分为多路由多页面的文档系统
- 不接入图片、视频或外部知识库

## 现状问题

### 内容过期

帮助中心未包含以下已上线能力：

- POS 页的筛选、分页、仅看我的设备、设备分类、认领、无权限提示
- License 备份与导入
- POS 数据库备份恢复
- 数据库配置页面与 SQL 模板执行
- WAR 包管理
- 个人中心
- 移动设备的图片上传、负责人设置、卡片/列表视图
- Linux 页面中的版本信息等细分能力

### 结构老化

- 左侧目录直接列出所有章节，模块增加后会越来越冗长
- 右侧内容为连续长文，缺少概览层
- 用户无法先按“模块”收敛范围，再按“任务”定位

### 展示单一

- 页面缺少总览区和功能摘要
- 内容块视觉统一度不够，重点信息不突出
- 移动端和窄屏场景下可读性较弱

## 设计方案

采用单页双层导航结构，保留现有 `/help` 路由。

### 第一层：顶部模块切换

顶部展示帮助模块 Tab，用户先选择当前要查看的模块：

- 快速开始
- POS 设备
- 移动设备
- Linux 配置
- 数据库配置
- WAR 包管理
- 工作台
- 管理中心
- 个人中心

第一层的职责是缩小内容范围，减少左侧目录过长的问题。

### 第二层：模块内目录

进入模块后，主体区域采用左右布局：

- 左侧：仅显示当前模块的二级目录
- 右侧：显示当前模块的详细内容卡片

二级目录通常包括：

- 模块概览
- 常用入口
- 常见流程
- 关键功能
- 注意事项

根据模块复杂度，也可以细分为多个具体章节。

### 内容展示

右侧内容不再是单一长文，而是按 section 卡片展示。每张卡片由结构化信息组成：

- 标题
- 场景说明
- 适用角色
- 入口路径
- 核心能力
- 操作步骤
- 注意事项

同时保留列表、表格、提示块等常见信息表达形式。

## 交互设计

### 模块切换

- 默认展示“快速开始”
- 点击顶部模块 Tab 后切换当前模块
- 切换模块时，二级目录和正文内容同步更新
- 默认激活当前模块的首个 section

### 模块内定位

- 点击左侧目录滚动到右侧对应 section
- 随滚动高亮当前 section
- 内容区域独立滚动，不影响页面整体导航

### 阅读辅助

- 模块头部展示一句简介和几个能力摘要
- 首页和每个模块顶部显示“适用对象 / 关键能力 / 相关入口”
- 长步骤内容使用有序列表
- 状态说明和风险说明使用高亮提示块

## 视觉设计

帮助中心整体风格保持与现有系统一致，但比现在更强调信息组织：

- 顶部增加明显的模块切换区
- 当前模块的概览区域使用大卡片样式
- 左侧目录使用浅色背景与激活态
- 右侧 section 卡片增强层次感和边距
- 用不同色块区分“入口 / 步骤 / 注意事项 / 角色”

布局要求：

- 桌面端为双栏布局
- 窄屏下顶部模块横向滚动，左侧目录下沉到内容上方
- 避免固定高度带来的内容截断

## 技术方案

仍在 `frontend/src/pages/HelpPage.jsx` 中实现，避免无谓拆分。

核心结构建议：

- 使用结构化的帮助数据源定义模块与 section
- 通过 `activeCategory` 管理当前模块
- 通过 `activeSection` 管理当前模块内的高亮 section
- 使用 `ref` 或 `id` + 滚动容器实现模块内定位

建议增加的视图层元素：

- `hero` 概览区
- 顶部 category tabs
- 左侧 section nav
- 右侧内容卡片
- 提示块与标签块

## 测试与验证

重点验证：

- `/help` 正常访问
- 顶部模块切换是否同步刷新左侧目录与正文
- 左侧目录点击是否正确滚动到对应 section
- 默认激活逻辑是否正确
- 所有新增模块内容是否可见
- 页面在窄屏下布局是否可读
- 构建是否通过

## 风险与应对

### 风险 1：内容量过大导致组件臃肿

应对：
- 保持统一的数据结构
- 抽取少量通用渲染辅助函数
- 不在本次提前拆成多个文件，避免过度工程化

### 风险 2：滚动定位在独立容器中不稳定

应对：
- 使用独立容器下的 section 定位逻辑
- 切换模块时主动重置激活 section 与滚动位置

### 风险 3：帮助内容继续增长后维护成本上升

应对：
- 统一每个模块的内容模板
- 将帮助内容组织成清晰的数据对象，便于后续继续追加

## 结论

本次帮助中心改版采用单页双层导航方案，在不改变用户入口的前提下，把帮助页面升级为适合当前产品体量的“模块索引 + 结构化教程”页面。它兼顾上线成本、交互清晰度与后续扩展性，是当前阶段最合适的折中方案。
