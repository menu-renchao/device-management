# 平台业务整改设计

**日期**: 2026-03-12

## 背景

当前设备管理平台把资产扫描、资产借用、远程运维、数据库执行、审批工作台放在同一产品里，但权限边界、入口组织和数据模型没有同步收敛，导致三类问题同时存在：

1. 高风险能力和普通资产浏览共享不一致的权限模型。
2. 同一业务流程被拆成多套前后端实现，造成入口重复和规则漂移。
3. 运维默认值和初始化策略把测试便捷性放在了安全边界之前。

这轮整改的目标不是做局部补丁，而是把“权限边界、审批流程、工作台入口、借用模型”一起收口，形成一版可持续演进的产品骨架。

## 目标

1. 所有扫描、Linux 运维、数据库配置与执行能力统一落在受控权限体系下。
2. 工作台成为唯一审批入口，普通用户默认看到自己的申请和借用，而不是审批视角。
3. POS 借用与移动设备借用统一为单一领域模型和统一接口。
4. 默认账号、默认口令、前端硬编码凭据全部移除。
5. 保持现有用户、设备、借用历史数据可迁移，不通过“清库重建”完成整改。

## 非目标

1. 本轮不拆分成多个独立服务，仍保留单体前后端工程结构。
2. 本轮不重做视觉设计系统，只做与流程、权限、信息架构直接相关的前端调整。
3. 本轮不替换 SQLite，也不引入外部权限服务。

## 方案选择

### 方案 A：仅做安全修补

只处理鉴权缺口、默认口令和默认管理员。

优点：
- 风险最小，交付最快。

缺点：
- 入口重复、借用模型重复、工作台默认流程错误仍然保留。
- 下一轮仍要再拆一遍前后端逻辑。

### 方案 B：安全修补 + 入口收敛

在方案 A 上统一审批入口和工作台默认视图，但保留 POS / 移动设备两套借用模型。

优点：
- 体验和安全都能明显改善。

缺点：
- 后端数据模型没有统一，未来规则变更仍需双改。

### 方案 C：彻底整改

同时完成权限收口、审批入口收敛、借用领域模型统一、工作台查询统一、初始化策略收紧。

优点：
- 业务骨架一次性理顺，后续扩展成本最低。

缺点：
- 涉及表结构迁移、接口契约调整和较大回归面。

**结论**：采用方案 C。

## 目标架构

### 1. 权限边界

后端新增统一的资产访问判定能力，负责判断某个用户是否可以对某个资产执行某个动作。最小角色集合保持为：

- `admin`
- `owner`
- `occupier`

动作按风险级别划分：

- `scan:view`
- `asset:borrow:approve`
- `linux:read`
- `linux:write`
- `db:read`
- `db:write`

所有 Linux、DB、日志、配置、备份、升级、扫描详情接口都统一经过这一层，而不是在 handler 中散落手写判断。

### 2. 借用领域模型

引入统一表 `borrow_requests`，字段至少包括：

- `id`
- `asset_type`
- `asset_id`
- `merchant_id`
- `requester_id`
- `approver_user_id`
- `status`
- `purpose`
- `end_time`
- `processed_by`
- `processed_at`
- `rejection_reason`
- `created_at`
- `updated_at`

其中：

- POS 资产通过 `asset_type=pos` + `merchant_id` 标识。
- 移动设备通过 `asset_type=mobile` + `asset_id` 标识。

原有 `DeviceBorrowRequest` 和 `MobileBorrowRequest` 只在迁移期用于数据回填，迁移完成后从运行路径中移除。

### 3. 工作台信息架构

工作台成为统一入口，固定包含四类视图：

- 我的申请
- 我的借用
- 我的设备
- 系统通知

审批视图只在用户具备待审批数据时显示。默认 tab 调整为“我的申请”。独立 `/borrow-approval` 页面及其重复逻辑删除。

### 4. 初始化与默认值

后端不再自动创建 `admin/admin123`。改为：

- 读取显式环境变量初始化管理员。
- 若数据库内无管理员且未提供初始化变量，则服务启动失败并输出初始化指引。

前端 Linux / DB 连接表单不再预置真实用户名密码，默认空值，仅保留安全的端口和数据库类型等非敏感默认值。

## 数据流

### 借用申请

1. 用户在 POS 或移动设备页面发起借用。
2. 前端统一调用新的借用申请接口。
3. 后端根据资产类型解析审批人，并写入 `borrow_requests`。
4. 工作台“我的申请”和“待我审批”都从统一借用查询接口读取。

### 审批

1. 审批人在工作台中看到统一待审批列表。
2. 批准或拒绝后调用统一审批接口。
3. 后端统一更新借用申请状态，并同步写入 POS 占用或移动设备占用状态。
4. 通知中心写入一条统一通知。

### 运维访问

1. 前端进入 Linux / DB 页面时只做体验层提示，不作为最终权限判定。
2. 后端接口先经过统一资产权限判断。
3. 只有通过校验的用户才能读取连接状态、日志、配置、备份、执行历史等敏感信息。

## 迁移策略

### 数据迁移

1. 新增 `borrow_requests` 表。
2. 从 `device_borrow_requests` 和 `mobile_borrow_requests` 读取历史数据并映射到新表。
3. 为统一工作台和审批接口切换到新表读取。
4. 保留旧表一段兼容期，但不再写入。

### 接口迁移

新增统一接口：

- `GET /api/borrow-requests`
- `POST /api/borrow-requests`
- `POST /api/borrow-requests/:id/approve`
- `POST /api/borrow-requests/:id/reject`

旧接口在迁移期内部转发到新服务，前端切换完成后再移除。

## 错误处理

1. 无权限统一返回 `403`，不再用“未连接”掩盖权限错误。
2. 未初始化管理员时，后端启动阶段直接失败，避免不安全默认值悄悄上线。
3. 数据迁移失败时服务不切换新路径，避免半迁移状态下新旧逻辑并存。
4. 工作台统一接口对未知资产类型返回明确错误，防止脏数据进入前端。

## 测试策略

### 后端

1. 为统一权限服务补充管理员、负责人、借用人、无权限用户四类单元测试。
2. 为统一借用服务补充 POS / mobile 两种资产类型的申请、审批、拒绝测试。
3. 为数据迁移补充旧表到新表的映射测试。
4. 为扫描、Linux、DB 高风险接口补充集成测试，验证未授权不可访问。

### 前端

1. 构建检查确保路由和页面依赖关系调整后仍可编译。
2. 验证工作台默认落在“我的申请”。
3. 验证无审批数据或无审批权限的用户不会再看到重复审批入口。
4. 验证 Linux / DB 页面默认不带敏感凭据。

## 风险与缓解

1. 借用模型迁移可能影响旧数据展示。
缓解：先做双写禁用、单读切换前的数据校验脚本。

2. 统一权限后可能暴露前端以前依赖的越权调用。
缓解：先补后端测试，再逐个修正前端调用。

3. 删除独立审批页可能影响已有收藏链接。
缓解：旧路由重定向到 `/workspace?tab=approvals`。

## 交付结果

整改完成后，平台应满足以下结果：

1. 扫描、运维、数据库能力全部需要明确授权。
2. 工作台成为统一业务入口，不再存在重复审批页面。
3. 借用审批规则只有一套实现，POS 与移动设备共享同一流程骨架。
4. 系统不存在硬编码真实凭据和默认管理员后门。
