# POS 设备管理页数据库配置功能设计

## 1. 背景与目标

当前系统在 POS 设备管理侧已具备扫描、设备属性、借用、Linux 运维等能力，但缺少“面向设备数据库的可配置 SQL 执行”能力。  
本设计目标是在 POS 设备管理中新增一个可审计、可控风险、支持权限隔离的数据库配置模块，满足产品经理提出的以下核心诉求：

- 在 POS 设备维度提供数据库配置能力，Linux 与 Windows 设备均可用。
- SQL 模板以列表方式管理，支持表格外新增、搜索、复选、编辑、删除、执行。
- 普通用户可新增模板、可编辑/删除自己创建的模板、可执行可见模板。
- 管理员拥有全部权限。
- 支持“多条 SQL 组合”执行，并输出逐条结果。

## 2. 设计范围

### 2.1 本期范围（In Scope）

- 新增“数据库配置”页面与入口（POS 设备列表按设备进入）。
- 新增设备数据库连接信息管理（首版仅 MySQL）。
- 新增全局 SQL 模板库管理（CRUD + 搜索 + 复选 + 执行）。
- 新增高风险 SQL 默认拦截与管理员强制执行机制。
- 新增执行审计（任务级 + 语句级明细）。

### 2.2 非本期范围（Out of Scope）

- 平台后端以外的执行链路（例如 SSH 到设备后再执行 DB 命令）。
- 多数据库类型支持（PostgreSQL、SQL Server）。
- 多设备批量执行（本期仅“单设备内执行多个模板”）。
- SQL 智能改写、语法高亮、版本回滚。

## 3. 已确认的关键决策

1. 执行链路：平台后端直连目标数据库执行（方案 A）。
2. 数据库类型：首版仅支持 MySQL。
3. 执行范围：仅当前设备，不做多设备批量。
4. 连接管理：按设备维护连接信息，并持久化。
5. 模板可见性：全局模板库；普通用户可见并可执行他人模板。
6. 模板编辑权限：普通用户仅能编辑/删除自己创建模板；管理员可管理全部。
7. 执行策略：多 SQL 逐条执行，失败继续，最终汇总。
8. 设备权限前提：执行必须具备设备权限（管理员/负责人/借用人）。
9. 风险控制：高风险 SQL 默认拦截，仅管理员可强制执行。
10. 审计要求：记录完整执行审计明细。

## 4. 信息架构与页面交互

## 4.1 入口与路由

- 在 POS 设备列表操作区新增按钮：`数据库配置`。
- Linux 与 Windows 设备都显示该按钮。
- 新增路由：`/db-config/:merchantId`。
- 无权限用户点击入口时，前端提示并阻止进入；后端仍做强制鉴权。

## 4.2 页面结构

页面分为两大区域：

1. 设备数据库连接信息（设备私有）
2. SQL 模板库（全局）

### 连接信息区域字段

- 主机（host）
- 端口（port，默认 3306）
- 数据库名（database）
- 用户名（username）
- 密码（password，默认掩码）
- 操作：测试连接、保存
- 辅助信息：最后更新人、最后更新时间

### 模板列表区域（表格）

- 表格外操作：`新增`
- 工具区：`搜索框（名称/备注）`、`复选框选择状态`
- 列：复选框、名称、SQL（摘要/可展开）、备注、创建人、更新时间、操作
- 操作列：编辑、删除、执行

## 4.3 模板编辑交互

- 新增/编辑通过弹窗完成。
- 字段：
  - 名称（必填）
  - SQL（必填，支持多条组合）
  - 备注（选填）
- 保存校验：
  - 名称非空
  - SQL 非空且可解析出至少 1 条有效语句

## 4.4 执行交互

- 支持两类执行动作：
  - 行内单模板执行
  - 勾选多个模板后统一执行（同一设备内）
- 执行前使用 `toast.confirm` 二次确认：
  - 显示目标设备
  - 显示模板数量
  - 提示执行策略（失败继续）
- 执行中展示进度信息（当前模板、当前语句、成功失败计数）。
- 执行后展示汇总与逐条明细（支持复制结果）。

## 5. 权限模型

采用“双重权限判定”：设备权限 + 模板权限。

## 5.1 设备权限（Device Scope）

- 可访问与执行角色：
  - 管理员
  - 设备负责人
  - 当前借用人（借用未过期）
- 与现有 Linux 权限模型保持一致，统一判定口径。

## 5.2 模板权限（Template Scope）

- 管理员：可增删改查并执行全部模板。
- 普通用户：
  - 可新增模板
  - 可编辑/删除自己创建模板
  - 可查看并执行全局模板

## 5.3 连接信息维护权限

- 管理员、负责人、借用人均可维护设备连接信息。
- 非上述角色禁止读写连接配置。

## 6. 功能需求详述

## 6.1 设备连接配置管理

### 功能点

- 查询当前设备连接配置。
- 新增/更新连接配置。
- 测试连接可达性和认证有效性。

### 校验规则

- 必填：host、port、database、username、password。
- 端口范围：1~65535。
- 测试连接失败时，不允许进入 SQL 执行流程。

## 6.2 SQL 模板管理

### 功能点

- 模板创建、更新、删除、详情查询、列表查询。
- 列表支持关键词搜索（名称、备注）。
- 列表支持复选与全选。

### 模板内容规则

- 支持多条 SQL 组合，推荐用分号分隔。
- SQL 原文保留，不自动改写。
- 列表中默认仅显示摘要，避免表格过长。

## 6.3 SQL 执行

### 执行规则

- 同设备同一时刻仅允许一个执行任务（设备级互斥）。
- SQL 逐条执行，失败继续。
- 最终状态：
  - success（全部成功）
  - partial_failed（部分失败）
  - failed（连接失败或全部失败）

### 返回结果

- 任务级：总数、成功数、失败数、总耗时、状态。
- 语句级：语句序号、执行状态、耗时、错误信息。

## 7. 风险控制与安全设计

## 7.1 高风险 SQL 检测

默认识别并拦截：

- DROP
- TRUNCATE
- 无 WHERE 的 DELETE
- 无 WHERE 的 UPDATE

### 处理策略

- 普通用户：命中即拒绝执行。
- 管理员：可选择“强制执行”，且必须填写原因。

## 7.2 凭据安全

- 密码字段仅密文保存（`password_encrypted`）。
- 前端接口不返回明文密码。
- 日志禁止打印连接密码和完整连接串。

## 7.3 防误操作

- 所有删除和执行动作均走确认弹窗。
- 执行按钮具备 loading 与防重复提交机制。

## 9. API 设计

统一前缀：`/api/db-config`，统一使用 JWT 认证。

## 9.1 连接配置接口

- `GET /connections/:merchantId`：获取设备连接配置（脱敏）
- `PUT /connections/:merchantId`：保存连接配置
- `POST /connections/:merchantId/test`：测试连接

## 9.2 模板管理接口

- `GET /templates`：模板列表（分页 + 搜索）
- `GET /templates/:id`：模板详情
- `POST /templates`：新增模板
- `PUT /templates/:id`：更新模板
- `DELETE /templates/:id`：删除模板

## 9.3 执行与审计接口

- `POST /execute`：执行模板（单设备）
- `GET /execute/:taskId`：查询任务结果
- `GET /execute/history`：审计列表（管理员全量，普通用户仅自己）

## 9.4 执行请求体建议

```json
{
  "merchant_id": "M123456",
  "template_ids": [101, 102],
  "force_execute": false,
  "force_reason": ""
}
```

## 10. 核心流程与数据流

1. 用户在 POS 列表进入某设备数据库配置页。
2. 系统校验设备权限，加载连接信息与模板列表。
3. 用户维护连接信息并测试连接。
4. 用户选择一个或多个模板，触发执行。
5. 后端执行前做：
   - 权限校验
   - 模板归档快照
   - SQL 风险检测
   - 连接检测
6. 执行器逐条运行 SQL，失败继续。
7. 任务结束后写入任务主记录与明细审计。
8. 前端展示汇总与逐条结果。

## 11. 异常处理规范

- `400`：参数错误、风险 SQL 被拦截、强制执行理由缺失
- `401`：未认证
- `403`：设备权限不足或模板编辑权限不足
- `404`：设备/模板/任务不存在
- `422`：数据库连接不可用
- `500`：服务内部错误

错误返回需包含可读 message，便于前端直接提示。

## 12. 测试与验收标准

## 12.1 功能验收

- Linux/Windows 设备均可见入口。
- 模板增删改查、搜索、复选、执行全部可用。
- 多 SQL 执行结果正确，失败继续策略生效。

## 12.2 权限验收

- 普通用户仅能编辑/删除自己创建模板。
- 普通用户可执行他人模板，但必须具备设备权限。
- 管理员具备全权限。

## 12.3 风控验收

- 普通用户命中高风险 SQL 必须拦截。
- 管理员强制执行需二次确认和原因，且可审计。

## 12.4 审计验收

- 审计记录覆盖任务级与语句级。
- 可追溯执行人、设备、模板快照、执行结果。

## 12.5 回归验收

- 不影响现有 Linux 运维页面与升级流程。
- 不影响扫描、借用、认领、用户管理等既有功能。
