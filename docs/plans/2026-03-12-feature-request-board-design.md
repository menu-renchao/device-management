# 意见收集板块设计

## 1. 背景与目标

当前系统已经具备登录、权限控制、通知、工作台和多类业务页面，但还缺少一个面向全体登录用户的公开意见收集入口。用户提出的新需求、改进建议和适配诉求，暂时没有统一沉淀位置，也无法通过用户反馈热度快速判断优先级。

本次目标是在现有系统内新增一个独立的“意见收集”板块，满足以下核心诉求：

1. 所有已登录用户都可查看和提交意见。
2. 所有已登录用户都可对意见点赞，每人每条意见仅可点赞一次，并可取消。
3. 列表支持按热度排行和按最新排序，便于识别高需求意见。
4. 管理员可维护意见处理状态，公开反馈软件适配进展。
5. 第一版聚焦核心闭环，不扩展评论、附件、复杂分类等非必要能力。

## 2. 已确认需求

本轮讨论已经确认以下范围：

- 可见范围：仅所有已登录用户，不对未登录访客开放。
- 点赞规则：每个用户对每条意见只能点赞一次，可取消。
- 处理流转：意见需要状态管理，不是纯展示留言墙。
- 状态维护权限：仅管理员可修改状态。
- 产品形态：采用方案 A，新建独立“意见收集”页面，由普通用户浏览和互动，管理员在该页完成状态维护。

## 3. 方案对比与选型

### 方案 A：独立意见广场页，管理员在卡片或详情中改状态

特点：

- 新增一个独立主页面作为公开意见广场。
- 普通用户与管理员都在同一页面消费内容。
- 管理员额外获得状态修改入口。

优点：

- 入口清晰，用户容易感知这是全员可见的需求池。
- 与“按点赞排行”的目标天然契合。
- 首版开发范围可控，前后端改动都比较集中。
- 后续扩展筛选、置顶、详情页都比较自然。

缺点：

- 如果后续管理动作很多，页面会逐步变重。

### 方案 B：意见广场与意见管理后台拆分

特点：

- 普通用户进入意见广场。
- 管理员进入独立管理页处理状态。

优点：

- 角色边界更清晰。
- 管理操作不会打扰普通用户浏览。

缺点：

- 首版开发量更大。
- 需要维护双入口和双页面状态。

### 方案 C：将意见模块挂入现有工作台 Tab

特点：

- 作为工作台中的一个子区域，不新增独立一级页面。

优点：

- 导航改动小。
- 入口统一。

缺点：

- 公开感和参与感不足。
- 热门意见的“榜单”属性会被弱化。
- 不利于后续独立扩展。

### 结论

采用方案 A：新增独立“意见收集”页面，所有登录用户可查看、提交、点赞，管理员可在同页维护意见状态。

## 4. 页面与交互设计

## 4.1 页面入口

位置建议：

- 前端主导航中新增一个“意见收集”入口。
- 路由建议新增为 `/feature-requests`。

权限：

- 使用现有 `PrivateRoute` 保护，仅登录用户可访问。

## 4.2 页面结构

页面建议由以下区域组成：

1. 顶部标题区
   - 页面标题：`意见收集`
   - 辅助说明：引导用户提交改进建议，并说明高热度意见会优先评估
2. 工具栏
   - 排序切换：`最热`、`最新`
   - 可选状态筛选：`全部`、`待评估`、`计划中`、`已完成`、`已拒绝`
   - `提交意见` 按钮
3. 意见列表区
   - 采用卡片列表展示
   - 每张卡片展示标题、内容摘要、提出人、发布时间、状态、点赞数
   - 当前用户已点赞时，点赞按钮显示选中态
4. 空状态区
   - 当列表为空时提示“还没有意见，欢迎提交第一条”

## 4.3 提交意见交互

建议使用弹窗表单，而非单独新建页面：

- 标题输入框
- 内容文本域
- 提交按钮

表单规则：

- 标题必填
- 内容必填
- 第一版不支持附件上传
- 第一版不支持标签分类

这样可以最大限度减少路由、状态管理和额外页面成本，同时贴近现有项目中常见的轻量操作模式。

## 4.4 卡片交互

普通用户卡片能力：

- 查看标题与内容
- 查看状态
- 查看点赞数
- 点赞 / 取消点赞

管理员额外能力：

- 可见状态修改入口
- 可将状态修改为：`待评估`、`计划中`、`已完成`、`已拒绝`

状态展示建议：

- `待评估`：中性色
- `计划中`：蓝色或强调色
- `已完成`：绿色
- `已拒绝`：灰色或弱提示色

## 5. 数据模型设计

## 5.1 主表：feature_requests

用途：存储意见主体。

建议字段：

- `id`
- `title`
- `content`
- `status`
- `created_by`
- `created_at`
- `updated_at`
- `like_count`

字段说明：

- `status` 使用后端枚举值：`pending`、`planned`、`completed`、`rejected`
- 前端文案映射为：`待评估`、`计划中`、`已完成`、`已拒绝`
- `created_by` 关联用户 ID
- `like_count` 为冗余字段，用于高效排序，避免每次列表查询都做聚合统计

## 5.2 点赞关系表：feature_request_likes

用途：记录用户对意见的点赞关系。

建议字段：

- `id`
- `request_id`
- `user_id`
- `created_at`

约束要求：

- 对 `(request_id, user_id)` 建立唯一索引

结果：

- 从数据库层面保证“每个用户每条意见只能点赞一次”
- 支持取消点赞时直接删除关系记录

## 6. 接口设计

接口统一放在认证后路由下，建议前缀为 `/api/feature-requests`。

## 6.1 获取意见列表

`GET /api/feature-requests`

支持参数：

- `sort=hot|latest`
- `status=`
- `page=`
- `page_size=`

返回字段建议包括：

- `id`
- `title`
- `content`
- `status`
- `author`
- `created_at`
- `updated_at`
- `like_count`
- `liked_by_me`

其中 `liked_by_me` 用于前端直接确定点赞按钮状态，避免额外查询。

## 6.2 创建意见

`POST /api/feature-requests`

请求体：

- `title`
- `content`

行为：

- 创建者取当前登录用户
- 初始状态固定为 `pending`
- 初始点赞数为 `0`

## 6.3 点赞意见

`POST /api/feature-requests/:id/like`

行为：

- 创建点赞关系
- 对主表 `like_count` 加一
- 如果该用户已经点过赞，接口按幂等成功处理更利于前端体验

## 6.4 取消点赞

`DELETE /api/feature-requests/:id/like`

行为：

- 删除点赞关系
- 对主表 `like_count` 减一
- 如果未点赞，接口可按幂等成功处理

## 6.5 修改状态

`PUT /api/feature-requests/:id/status`

权限：

- 仅管理员

请求体：

- `status`

后端校验：

- 必须是 `pending`、`planned`、`completed`、`rejected` 之一

## 7. 权限与行为边界

普通用户：

- 可查看全部意见
- 可创建意见
- 可点赞和取消点赞
- 不可修改状态
- 第一版不可编辑或删除意见

管理员：

- 拥有普通用户全部能力
- 可修改任意意见状态

第一版特意不纳入以下能力：

- 评论
- 回复
- 置顶
- 编辑意见
- 删除意见
- 附件上传
- 分类标签

这样可以确保本次仅围绕“收集 -> 点赞聚合 -> 管理员公开反馈状态”这一条主线交付。

## 8. 数据流设计

## 8.1 浏览列表

1. 用户进入“意见收集”页面
2. 前端按默认排序 `hot` 请求列表
3. 后端返回分页数据，并带上当前用户的 `liked_by_me`
4. 前端渲染卡片列表

## 8.2 提交意见

1. 用户点击 `提交意见`
2. 前端弹出表单并校验标题和内容
3. 前端提交创建请求
4. 后端写入意见主表，状态默认为 `pending`
5. 前端提示成功并刷新列表，或将新意见插入当前列表

## 8.3 点赞/取消点赞

1. 用户点击点赞按钮
2. 前端发起点赞或取消点赞请求
3. 后端更新点赞关系和 `like_count`
4. 前端更新卡片上的点赞数和按钮状态

## 8.4 管理员改状态

1. 管理员在卡片上点击状态操作入口
2. 前端提交新的状态值
3. 后端校验管理员权限与状态值合法性
4. 更新成功后，前端仅刷新当前卡片或重新拉取列表

## 9. 异常处理

需要覆盖的异常场景：

- 标题为空
- 内容为空
- 意见不存在
- 非管理员尝试修改状态
- 状态值非法
- 重复点赞
- 取消未点赞的意见
- 列表为空

处理策略：

- 前端优先做基础必填校验
- 后端做最终校验和权限兜底
- 点赞/取消点赞尽量做幂等，减少竞态和重复点击问题
- 操作失败后保留当前页面状态，并展示明确错误提示

## 10. 测试策略

## 10.1 后端测试

重点覆盖：

- 创建意见成功
- 列表按最新排序正确
- 列表按热度排序正确
- 同一用户不能重复点赞
- 取消点赞后 `like_count` 正确回退
- 非管理员修改状态返回无权限
- 管理员修改状态成功

## 10.2 前端验证

重点覆盖：

- 登录用户可以进入新页面
- 页面默认正确拉取并展示列表
- 排序切换生效
- 提交意见表单校验生效
- 点赞和取消点赞按钮状态正确变化
- 管理员能看到状态修改入口
- 普通用户看不到状态修改入口

## 12. 结论

本次设计将意见收集能力作为系统内一个新的公开协作入口落地，重点是建立一个简单、透明、可排序、可反馈状态的需求池，而不是一次性做成复杂社区模块。

第一版通过“全员可见 + 点赞热度 + 管理员状态维护”的组合，已经能够支撑高需求意见的显性化和开发适配决策。后续若使用频率高，再考虑补充评论、分类、置顶或管理员管理增强能力。
